Elderly Monitoring System — Architecture Approaches

3 signals: Android accelerometer · CCTV outside · Alexa timer  |  Alerts: Telegram + Dashboard + Phone call

A

Local Server (Ubuntu) + Telegram

Android app → FastAPI (Ubuntu) → rules engine
Alexa IFTTT → FastAPI (Ubuntu) → rules engine
kensho-local CCTV → Telegram bot → same rules engine
rules engine → Telegram alerts + Twilio call

Pros

  • Zero cloud cost
  • Reuses kensho-local CCTV
  • Telegram already set up
  • Fast to build

Cons

  • Ubuntu must be on 24/7
  • No access outside home WiFi (unless Tailscale)
B

Cloud (GCP Free Tier) + Telegram

Android app → Cloud Run (GCP) → Firestore
Alexa IFTTT → Cloud Run (GCP) → Firestore
kensho-cctv-telegram → Cloud Run (GCP) → Firestore
Cloud Scheduler → rules engine → Telegram + Twilio

Pros

  • Always available, Ubuntu off is fine
  • Accessible from anywhere
  • GCP free tier covers this load

Cons

  • More setup (Cloud Run + Firestore)
  • kensho must be cloud version
C

Hybrid — Local collect, Cloud rules (recommended)

Android app → local FastAPI → Google Sheets (log)
Alexa IFTTT → Google Sheets (direct)
kensho-local → Google Sheets (direct)
Apps Script trigger → rules engine → Telegram + Twilio

Google Sheets as the central data store — free, no infra, familiar. Apps Script runs the rules every 10 min. Android app posts to local FastAPI which writes to Sheets.

Pros

  • Zero cost
  • Google Sheets = instant dashboard
  • Same pattern as TrackWise Phase 5
  • Rules engine = Apps Script (no new infra)

Cons

  • Android app needs Ubuntu on to post data (or posts directly to Sheets API)

My recommendation: Option C — Google Sheets as data hub, zero cost, reuses TrackWise pattern. Android app posts directly to Sheets API (no Ubuntu needed).
